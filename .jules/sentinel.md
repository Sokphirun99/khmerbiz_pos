## 2024-05-30 - Insecure PIN Storage Fixed with Iterative Salted Hashing
**Vulnerability:** The application was storing user PINs using a single-iteration SHA-256 hash without a salt. For short secrets like 4-6 digit PINs, this allows an attacker to trivially generate a rainbow table or perform an instantaneous brute-force attack to reverse the PIN if they gain access to the database.
**Learning:** Using standard cryptographic hash functions without a salt and without key stretching (e.g., thousands of iterations) is fundamentally insecure for short secrets like PINs. Even though the database is encrypted, relying on encryption alone without proper hashing creates a single point of failure (if the DB key is extracted, the PINs are immediately compromised).
**Prevention:** Always use a random, unique salt per user (such as their UUID) and an iterative hashing algorithm (like PBKDF2, bcrypt, Argon2, or manually implementing key stretching by iterating SHA-256 thousands of times) to significantly slow down brute-force attacks against short PINs and passwords.
